# Assistant Guardrails

These are the working rules I should treat as binding when helping on this course work.

## Highest Priority Rules

1. Do not invent formats.
2. Do not rename files, projects, solutions, or zip files creatively.
3. Do not replace lecturer conventions with personal style preferences.
4. Do not use unsupported tooling or unsupported submission structure.
5. Do not leave generated folders in the final hand-in.
6. Do not ignore the coding standard details just because the code still compiles.

## Coding Guardrails

- Use the lecturer naming conventions exactly.
- Use braces for statement bodies.
- Use tabs for indentation.
- Keep one return statement per method when implementing course-style solutions.
- Avoid duplicated code.
- Keep method and type names meaningful and consistent with the standards.
- Respect parameter prefixes: `i_`, `o_`, `io_`.
- Respect member prefixes: `m_`, `k_`, `s_`, `r_`, `sr_`.
- Respect enum prefix `e`.
- Keep boolean names positive and use `!` for the negative case instead of negative boolean names.

## Solution Structure Guardrails

- Prefer a blank solution with separate projects added into it.
- Keep the solution structure compatible with `Visual Studio for Windows`.
- Keep the folder tree ready for zipping without cleanup surprises.

## Submission Guardrails

- Treat naming formats as exact, not approximate.
- Use spaces where the lecturer examples use spaces.
- Remove `bin`, `obj`, `.git`, and `packages` before final packaging.
- Keep the zip structure clean and predictable.
- If emailing, use the exact lecturer subject prefixes.

## Review Guardrails

Before saying a submission is ready, I should verify:

- naming
- folder cleanliness
- Visual Studio compatibility
- coding style compliance
- final zip readiness

If any of those fail, the work is not submission-ready yet.
